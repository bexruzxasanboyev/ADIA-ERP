## 2.5 ISHLASH SXEMASI — TIZIM MOHIYATI

### A. Zanjir va so'rov oqimi (katta rasm)

```
                    POSTER POS (manba haqiqat: savdo, qoldiq, retsept, поставка)
                          │ sinxron (read; write-back dry-run)
                          ▼
  Ta'minotchi ──Поставка──▶ HOMASHYO OMBORI ──xom-ashyo──▶ SEXLAR/ОТДЕЛЛАР ──tayyor──▶ MARKAZIY SKLAD ──▶ DO'KONLAR ──▶ Mijoz
   (Poster'da)            «Основной склад»(11)             (Наполеон 114…)              (15)              (Кукча 6…)
                               ▲                            ▲       │
                               │      ◀──SO'ROV (yuqoriga)──┘       ▼
                               └────── SO'ROVLAR ◀────────  sex_storage (з/г bufer, min/max)
                                                            (Qaymoq skladi 205 → krem barcha sexlarga)

  QOIDA: mahsulot PASTGA oqadi, so'rov YUQORIGA (parent_id) yoki YONGA (semi →
  uni ishlab chiqaruvchi отдел — products.workshop_location_id, "producer pin").
```

### B. Bitta so'rovning to'liq hayoti (data + darvozalar)

```
 Do'kon «10 kg Napoleon» dedi
   │  INSERT replenishment_requests (status=NEW, requester=do'kon, target=NULL)
   ▼
 MARKAZ DARVOZASI: cron store-so'rovga TEGMAYDI — markaz boshlig'i qaror qiladi
   │  /fulfill: bor 4 kg → movement(markaz→do'kon)+shipment_movement_id,
   │            original CLOSED+closure NULL (= "Yo'lda", REZERV)
   │            yetmagan 6 kg → YANGI so'rov (origin='shortfall', batch bir xil)
   ▼
 ОТДЕЛ DARVOZASI (d-klass): production_location_id=COALESCE(po.location, workshop)
   │  to'lguncha CHECK_PRODUCTION_INPUT'da TURADI (fulfiller_accepted_at IS NULL)
   │  отдел «Qabul» (accept-production) → stamp + O'SHA tranzaksiyada birinchi hop:
   │     з/г bufer (sex_storage) birinchi yeyiladi → yetmagan komponentga:
   │        semi → producer-otdelga SUB-SO'ROV (parent_request_id, daraxt!)
   │        raw  → PURCHASE ORDER (draft → boshliq+skladchi 2 imzo → received)
   ▼
 zayafka (production_orders, stage_role final/zagatovka) → «Tayyor» (PATCH done)
   │  finishProductionOrder: BOM yeyiladi + chiqim markazga + so'rov DONE_TO_WAREHOUSE
   ▼
 markaz «Qabul qildim» → do'konga jo'natadi → DO'KON «Qabul qilish» (brak bilan):
   │  receive: yaroqli+brak ≤ SHIPPED (movement qty!); brak markazga qaytadi
   │  closure_reason yoziladi → YOPILDI; bola yopilsa → waiter-rootlar advance (daraxt)
   ▼
 HOMASHYO YO'LI (agar target=raw pin bo'lsa): boshliq «Qabul» → "Poster kutilmoqda"
    → Poster'da Поставка qo'shiladi → stockSync keltirgach cron AVTO-jo'natadi
```

### C. Data modeli mohiyati (6 ta yuk ko'taruvchi jadval)

```
 locations (daraxt: parent_id=so'rov yo'li) ─┬─ location_flows (M:N fizik oqim)
                                             └─ har biriga manager (user)
 products ── workshop_location_id (kim ishlab chiqaradi — "producer pin"ning asosi)
          └─ recipes (BOM, stage: base/decoration; semi rekursiv ochiladi)
 stock (location,product) PK — qty/min/max; min'dan past → scan so'rov ochadi
 replenishment_requests — MARKAZIY OBYEKT:
    status(10) → pipeline_stage(5, SQL+TS ko'zgu) → UI bucket(3: YANGI/JARAYONDA/TAYYOR)
    parent/root/depth/origin (daraxt) · fulfiller_accepted_at (darvozalar kaliti)
    shipment_movement_id (jo'natilgan MIQDOR shu yerdan!) · journey (mini-xarita, derive)
 stock_movements — har harakat ATOMAR (kamayish+oshish+audit bitta tranzaksiya);
    replenishment_id bilan so'rovga bog'lanadi
 production_orders / purchase_orders — zayafka va xarid; request ularga id orqali
    bog'lanadi (purchase M:N ham bor: replenishment_purchase_orders)
```

**Uch oltin qoida (hamma narsa shulardan kelib chiqadi):**
1. Hech narsa odam tasdig'isiz oldinga yurmaydi (3 darvoza: markaz, отдел, homashyo).
2. Bitta (mahsulot, so'rovchi)ga bitta ochiq so'rov — dublikat o'rniga daraxt/waiter.
3. Hisob-kitob har doim HAQIQIY jo'natilgan miqdorga (movement), so'ralganga emas.
